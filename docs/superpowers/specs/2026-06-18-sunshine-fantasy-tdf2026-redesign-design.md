# Sunshine Fantasy — TdF 2026 Redesign (Design Spec)

**Date:** 2026-06-18
**Status:** Approved for planning
**Source of truth:** `design_handoff_tdf2026/README.md` (full visual + interaction spec) and `design_handoff_tdf2026/Standings.dc.html` (working prototype).

---

## 1. Goal

Recreate the high-fidelity "Sunshine Fantasy" redesign — a minimal, editorial, mobile-and-desktop-friendly fantasy-cycling app for the Tour de France 2026 — as a standalone **React + Vite single-page app**. The scoring model is unchanged: six players each draft three pro riders; a team's score is the **sum of its three riders' GC times**; the **lowest total wins**.

The handoff HTML is the visual + interaction source of truth. Exact tokens (colors, type, spacing, radii, shadows) are listed in the handoff README §"Design Tokens" and embedded in the prototype's inline styles — they are lifted directly, not re-derived.

## 2. Decisions (resolved during brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Stack | **React + Vite SPA** (new `web/` dir) | Prototype is already a React-shaped state machine; the interactions are the product and map 1:1 to components. Streamlit fights this design. |
| Data, phase 1 | **Placeholder-first** | No live 2026 race until July 4; real league picks don't exist yet. Build + verify the UI now behind a clean seam. |
| Language | **JavaScript (JSX), no TypeScript** | Matches prototype 1:1; lean toolchain. |
| Styling | **Inline styles + CSS custom properties** | Handoff calls the HTML the source of truth; lift its values directly rather than re-deriving into Tailwind/CSS-modules. |
| Routing | **No router library**; state-driven screens | Matches README state model. `?race=` query param preserved via `URLSearchParams`. |
| Draft persistence | **Ephemeral local state** (resets on reload) | Matches prototype ("no global store needed"); no auth/backend in this phase. |
| Coexistence | New `web/` dir; existing Streamlit app untouched | No risky cutover. Flip links to the new host only when ready. |
| Hosting | Static host (Vercel recommended) — **documented, not executed** this phase | User pushes the button when ready. |

## 3. Non-goals (this phase)

- The Python data generator / live procyclingstats wiring (see §7 — designed now, built in the live-data phase).
- Real 2026 league picks and real rider/stage data.
- Draft persistence, auth, or any backend.
- Standing up the Vercel deployment (documented only).
- Removing or modifying the existing Streamlit app.

## 5. State model

All UI state lives in `App.jsx` (lifted from the prototype's `DCLogic`):

```js
screen       // 'standings' | 'stage' | 'team' | 'races'   (default 'standings')
sub          // null | 'draft' | { type:'rider', name }    (active sub-screen)
expanded     // { [teamName]: bool }                        (default { Nate: true })
team         // selected player on Team tab                 (default 'Aaron')
race         // race id, synced to ?race=                   (default 'tdf-2026')
draftPicks   // string[] max 3                              (seeded from prototype)
showMovement // bool global toggle                          (default true)
```

`accent` is **derived from the selected race** (not independent state): tdf-* → Tour Yellow, giro-* → Giro Pink, vuelta-* → Vuelta Red. `theme.js` maps the accent name to the four CSS-variable values, applied on a root wrapper as `--accent`, `--accent-tint`, `--accent-border`, `--accent-ink`.

### Transitions (match prototype exactly)
- Tab click → set `screen`, clear `sub`.
- Standings row click → toggle `expanded[teamName]` independently (multiple open allowed).
- Team rider card / Stage "your riders today" card → set `sub = {type:'rider', name}`.
- Team (your team) "Edit your picks" → set `sub = 'draft'`.
- Back link → `sub = null`.
- Draft add → fill next empty slot; blocked at 3 and on duplicates. Remove → empty that slot.
- Race card click → set `race` (updates `?race=`, swaps accent theme; full dataset swap is a live-data-phase behavior).
- `showMovement` off → hide all ▲/▼/hold indicators across every screen.

## 6. Screens (fidelity reference)

Each recreates the corresponding prototype block at the token values in the handoff. Summary of intent; the HTML governs exact pixels.

1. **Standings (default):** header (eyebrow, "Updated", title, stage progress bar, daily-recap callout) → column labels → 6 ranked rows (rank + movement arrow, name, Yellow Jersey badge on leader, 🐼 on last, riders subline, total time + gap, chevron; leader row has accent-tint bg + 3px accent left bar) → expand reveals 3 riders (GC `#n`, name, today's delta, cumulative time) → footer note.
2. **Stage:** stage header → winner callout (🏆 tile) → "League shake-up" movers list → "Your riders today" tappable cards (→ Rider profile).
3. **Team:** horizontal scrollable player chip row (selected = solid accent) → selected team header (name + You/Yellow Jersey/🐼 badges, total + gap line, standing line) → 3 rider cards (44px GC chip, name, pro team, time + gap, Today movement row) → "Edit your picks" button (your team only → Draft).
4. **Races:** "Choose competition" eyebrow → race cards (status dot, name + Viewing badge, dates + stages, status badge Live/Complete/Upcoming + note). Selecting sets `viewing`; full dataset+theme swap is the live-phase behavior.
5. **Draft (sub-screen):** back link → header + lock note → 3 pick slots (filled: numbered accent tile, name, team·role, × remove; empty: dashed) → "Available riders" pool (name, team, role pill, + add — accent when slots remain, disabled gray at 3/3) → count "N / 3 picked" (red until full, green at 3) → confirm button (disabled sand until full, then solid-accent "Save team").
6. **Rider profile (sub-screen):** back link → header (role · nationality · age; big name; pro team) → 3 stat cards (GC rank / GC time / to-leader gap) → "Recent form — last 3 stages" tiles → owner callout.

Emoji: **only** 🏆 (stage winner) and 🐼 (lanterne rouge). No other emoji.

## 7. Data seam — preserving the procyclingstats investment

**Principle: nothing useful is thrown away.** The existing Python data layer (`api_client.py` + `procyclingstats`, plus rosters in `races_config.py`) becomes the **generator** for the SPA's data, not dead code.

### Phase 1 (this build)
- `data/sampleData.js` holds the prototype's `data()`, `racesData()`, and `draftPoolData()` ported verbatim.
- All screens read through **`useLeagueData(raceId)`**, which returns `{ teams, races, draftPool, stage, movers, yourToday, meta }`. Components never import `sampleData` directly — the hook is the only seam.

### Phase 2 (live data — designed now, built later)
- A Python script (`scripts/generate_data.py`, reusing `api_client.fetch_fantasy_standings` / `fetch_stage_by_stage_data` and `races_config`) writes `web/public/data.json` for the active race.
- `useLeagueData` switches from importing `sampleData` to `fetch('/data.json')` — **one file changes**, no component touches.
- A GitHub Action cron regenerates `data.json` every ~10–15 min during the 3 weeks of the Tour (GC changes once per stage/day, so cron suffices; no always-on server needed).

### Data contract (`data.json` shape — the documented boundary)
Mirrors `sampleData.js` so the swap is transparent:

```jsonc
{
  "meta": { "raceId": "tdf-2026", "name": "Tour de France 2026",
            "stageNum": 11, "totalStages": 21, "progressPct": "52%",
            "updated": "2 min ago", "recap": "…" },
  "teams": [ { "name": "Nate", "rank": 1, "total": "129:57:19", "gap": "Leader",
               "move": 1, "leader": true, "last": false,
               "riders": [ { "name": "…", "gc": 5, "time": "43:15:36", "d": 0,
                             "proTeam": "…", "gapGC": "+3:31", "role": "Climber",
                             "nat": "Germany", "age": 25, "form": [6,5,4] } ] } ],
  "races": [ { "id": "tdf-2026", "name": "…", "dates": "…", "stages": 21,
               "status": "Live", "dot": "#F2C200", "note": "Stage 11 / 21" } ],
  "draftPool": [ { "name": "…", "team": "…", "role": "GC" } ],
  "stage": { "date": "…", "route": "…", "type": "…", "km": "…",
             "winner": "…", "winnerTeam": "…", "winnerTime": "…" },
  "movers": [ { "name": "Nate", "move": 1, "note": "Takes the yellow jersey" } ],
  "yourToday": [ { "name": "…", "place": "2nd", "gap": "+0:24", "note": "Holds GC #1" } ]
}
```

### Field provenance (what's available vs. derived) — recorded so phase 2 has no surprises
- **Directly from procyclingstats GC** (already fetched by `api_client.py`): rider `name`, `time`, GC `rank` (`gc`), `proTeam`/`team_name`, computed team `total` + `gap` + `rank`.
- **Derived from a daily snapshot diff** (not in a single GC fetch): team `move`, rider `d` (today's GC delta), `movers`, `yourToday` deltas. **Requires persisting a daily standings snapshot** to diff today vs. yesterday — flagged as a phase-2 dependency.
- **From stage page / supplementary scrape:** `stage.*` (route, type, km, winner, winnerTime), rider `form` (last-3 finishes), `nat`, `age`. Available from procyclingstats but via additional endpoints beyond the current GC fetch.

`sampleData.js` carries the full shape (including derived fields) so phase-1 UI is complete and the contract is concrete before any wiring.

- Side-by-side visual check against `Standings.dc.html` opened in a browser.
